Come funziona davvero: IPR come gate ex-ante

L’IPR (Identity / Integrity Primary Record) è un record di precedenza che vincola l’azione prima dell’esecuzione. La regola è semplice e misurabile: se il contesto non coincide con l’IPR, l’azione non passa.

Che cos’è un IPR

Un IPR è un record deterministico che descrive, prima dell’azione, gli elementi minimi che rendono un atto eseguibile: contesto, policy, limiti, responsabilità. Il record è verificabile tramite hash e regole di coerenza.

PASS / FAIL (gate tecnico)

AI JOKER-C2 esegue un pre-check: confronta contesto reale e contesto attestato nell’IPR. L’esito è binario:

  • PASS → l’azione può essere eseguita
  • FAIL → l’azione è bloccata (fail-closed)
PASS FAIL no implicit trust

Perché “una virgola diversa” = FAIL

La governance ex-ante funziona solo se la verifica è deterministica. Questo significa che una modifica anche minima nel contesto (configurazione, policy, vincoli, scope) produce un hash differente e quindi un mismatch.

  • IPR A → hash A
  • Contesto modificato (anche minimo) → hash B
  • A ≠ B → FAIL → blocco
determinismo verifica replicabile no “aggiustamenti” ex-post

Sequence flow (semplificato)

Soggetto responsabile
  → Create IPR
IPR-CORE
  → Validate / Hash
AI JOKER-C2
  → Pre-check (contesto/policy/config)
  → PASS: Allow → AI Target Execute
  → FAIL: Block (no output)
UNEBDO (opzionale)
  → Evidence Pack (audit)

Proprietà misurabili

  • Copertura pre-check: % azioni verificate ex-ante
  • Tempo di pre-check: ms/s
  • Blocchi ex-ante: azioni impedite
  • Ripetibilità: verifica identica su re-run

Cosa NON è questa tecnologia

  • Non è sorveglianza: nessun controllo sulla persona, solo sull’atto.
  • Non è fiducia: non si “crede” al sistema, lo si verifica.
  • Non è correzione dopo: se manca precedenza, l’atto non avviene.
