THREAT MODEL · PREVENZIONE

Sicurezza ex-ante: ridurre superficie d’attacco prima dell’azione

Questo modello non promette invulnerabilità. Promette un criterio operativo: l’azione sensibile passa solo se evidenza minima verificabile coincide (PASS). Altrimenti: FAIL.

Specifica gate (PASS/FAIL) Prova minima (Verify) Limiti dichiarati

Asset protetti (cosa vale davvero)

  • azioni sensibili su tool e sistemi (deploy, comandi, scrittura, invio)
  • dati ad alta criticità (credenziali, PII, segreti industriali, policy)
  • integrità della supply chain (config, versioni, dipendenze)
  • auditabilità: capacità di replicare “perché è successo”

Attaccanti (modello minimo)

Opportunista

sfrutta default permissivi, errori, confusione operativa

Supply-chain

inserisce drift: versioni, dipendenze, config, pipeline

Insider / coercizione

usa accessi legittimi per far passare azioni non autorizzate

Vettori tipici (AI/automation)

Tool abuse

  • azioni su file/db senza vincoli
  • comandi e deploy “in automatico”
  • privilege creep (permessi che crescono)

Prompt / instruction injection

  • istruzioni malevole camuffate
  • trucchi per bypassare policy
  • confusione tra dati e comandi

Data exfiltration

  • estrazione indiretta (log, output, strumenti)
  • leak di segreti (token/chiavi)
  • ricostruzione di contesto sensibile

Drift & mismatch

  • policy aggiornata ma runtime no
  • config cambiata senza tracciamento
  • dipendenze mutate (supply-chain)

Controlli preventivi (ex-ante)

Il gate riduce rischio imponendo “eseguibilità” come condizione tecnica. Non sostituisce security engineering: lo rende verificabile e bloccante quando manca evidenza.

Evidence check

IPR canonico + digest: se non combacia → FAIL

deterministico

Context fingerprint

se runtime/pipeline/config non coincide con atteso → FAIL

anti-drift

Policy fingerprint

se policy dichiarata ≠ policy applicata → FAIL

anti-bypass

Limits (scope)

limiti espliciti: “no surveillance”, “no custody”, “no auto-rights”. Se l’azione viola i limiti → FAIL.

scope control

Fail-closed default

niente fallback silenziosi. Se il check non è possibile, non si esegue.

default: block

Misure (KPI) collegate alla sicurezza

  • Block rate: quante azioni pericolose vengono bloccate prima dell’evento
  • Mismatch rate: drift policy/runtime rilevato ex-ante
  • Reproducibility: audit replicabili su richiesta di terzi
  • MTTR: tempo per riallineare canonico/policy e tornare in PASS

Prova immediata

La prova minima non è un PDF: è Verify. Se il sistema non riesce a provare coerenza su file pubblici, non è pronto a governare azioni reali.
